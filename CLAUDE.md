# 三份文档的角色与边界

## spec.md
回答：要解决什么问题、做哪些能力、不做哪些、什么算完成。
写：背景、目标用户（如备考雅思的大学生、商务出差人群）、能力清单（一句话一条）、
    非功能要求（延迟、并发、隐私）、对话与评测的设计骨架、Out of Scope。
不写：具体 prompt 文本 / API 参数名 / 模型名称 / 阈值数字 / 错误文案 / SDK 类型名
    （这些是实现细节，spec 改一次就过期，维护爆炸）

## tasks.md
回答：按什么顺序做、每步动什么文件。
- 5~15 个任务，每个能在一次专注会话内完成
- 每个任务标注：影响文件（如 stt/recorder.py、prompts/interview.txt、scenarios/restaurant.yaml）、
  依赖任务、参考资料定位（Azure 发音评测 API 文档章节、Whisper 参数表、精确到函数/行号都可以）
- 典型任务粒度示例：「接入 STT」「实现单场景对话循环」「加入发音评分」「生成课后报告」
- 最后一定有「接入主流程」(把 STT→LLM→TTS→评测→报告 串成端到端流水线) 
  + 「端到端验证」(真人完整跑一次面试场景，产出报告) 两个任务

## checklist.md
每一项必须可勾选、可观测，不许写「对话自然」「发音评测准确」「报告有用」。
- 把 spec 里被砍掉的具体值（延迟阈值、评分区间、错误文案、默认场景、模型名）放进来作为验收项
- 写法举例：
  - 「从用户说完话到 AI 开口，P50 延迟 ≤ 1.5s，P95 ≤ 2.5s（用 tests/latency_bench.py 测）」
  - 「输入音频 fixtures/hello.wav，发音评分返回 {accuracy, fluency, completeness} 三个 0-100 整数」
  - 「断网时屏幕显示文案 "网络连接中断，请检查后重试"，而非堆栈报错」
  - 「scenarios/ 目录下至少有 3 个 .yaml 场景文件：interview / restaurant / meeting」
  - 「跑完一次完整会话后，reports/ 目录生成 1 个 .md 报告，包含发音分、纠错条数、对话轮数」
- 至少一条端到端验收：「选择面试场景 → 进行 ≥5 轮对话 → 退出 → 在 reports/ 看到本次报告」
